# ASR-датасет: синтетическая русская речь

Дата: 2026-07-10.

Цель: собрать 60-80 русских аудиофраз из общего tool-calling датасета для проверки ASR перед agent loop. Пользователь не может записать голоса вручную, поэтому базовый набор делаем синтетически, а затем валидируем вручную.

## Вывод

Рекомендуемый recommended pipeline:

1. Взять 72 фразы из приватного approved tool-calling корпуса.
2. Синтезировать каждую фразу в 4 голосах, но в manifest выбрать один основной голос на строку, чтобы стартовый набор остался 72 аудио.
3. Сохранять чистый WAV как базу.
4. Детерминированно добавить фермерский шум отдельным post-processing шагом: `quiet`, `farm_light`, `farm_medium`.
5. Хранить golden transcript и entity-level разметку рядом с аудио.
6. Считать `WER`, `entity_token_error_rate`, `animal_tag_exact_match`, `utterance_exact_match`.

## Рекомендуемый сервис

### Вариант 1: Yandex SpeechKit

Почему подходит для текущего контура:

- российский стек и русские голоса;
- есть управляемый API синтеза;
- проще ожидать адекватное произношение русских чисел, дат и кириллицы;
- удобно использовать как baseline для “фермерских” коротких команд.

Риски:

- качество кличек, бирок `A-17`, `E-55`, RFID и препаратов нужно слушать вручную;
- может потребоваться нормализация текста перед TTS: `A-17` -> `а семнадцать`, `E-55` -> `е пятьдесят пять`, `RFID` -> `эр эф ай ди`;
- голоса могут звучать слишком “студийно”, поэтому шум нужно добавлять отдельно.

### Вариант 2: SaluteSpeech

Почему полезно проверить:

- сервис ориентирован на русский speech stack;
- поддерживает SSML и управление интонацией/скоростью/паузами;
- есть и TTS, и ASR, поэтому можно использовать для сравнительного sanity-check.

Риски:

- надо проверить доступность API/тарифов и удобство пакетной генерации;
- нужно вручную сравнить произношение доменных сущностей с Yandex.

### Вариант 3: Azure Speech

Почему полезно как контроль:

- зрелый Speech SDK;
- поддерживает `ru-RU`;
- хорошие инструменты SSML, голоса, batch/REST integration.

Риски:

- может быть менее естественная русская доменная речь, чем у локальных русских провайдеров;
- доступность и биллинг могут быть тяжелее для небольшого набора.

### Вариант 4: ElevenLabs

Почему можно использовать:

- сильное качество естественности;
- много голосов и быстрый API;
- может хорошо разнообразить голоса.

Риски:

- для ASR-теста Cattle Track важнее правильность русских доменных токенов, а не актерская естественность;
- произношение бирок, RFID, препаратов и смешанных латинско-цифровых кодов нужно особенно строго валидировать.

## Не рекомендую как основной путь

- Ручные онлайн-генераторы без API: плохо воспроизводятся и неудобны для 60-80 файлов.
- Только один TTS-голос: ASR-метрика будет слишком оптимистичной.
- Просить TTS “сгенерировать шум фермы”: шум станет недетерминированным и может менять транскрипт. Лучше чистый TTS + отдельная аудио-аугментация.
- Сразу клонировать реальные голоса без юридически чистых согласий. Для для текущего этапа хватит готовых голосов.

## Нормализация текста перед TTS

Golden transcript должен оставаться пользовательским:

```text
внеси эмбрион E-55 для 981
```

Но TTS input может быть произносительной версией:

```text
внеси эмбрион е пятьдесят пять для девятьсот восемьдесят один
```

Поэтому в будущий audio generation manifest стоит добавить поля:

- `golden_transcript` - что должен вернуть ASR;
- `tts_text` - что отправляем в TTS для правильного произношения;
- `normalization_notes` - что было изменено.

Для первых 72 строк `tts_text` можно сначала сделать равным `golden_transcript`, а после ручного прослушивания поправить проблемные коды.

## Шум фермы

Нужны отдельные фоновые WAV:

- вентиляция/двигатель;
- шаги/металл/ворота;
- животные на фоне;
- помещение/эхо.

Рекомендуемые уровни:

| Профиль | SNR | Назначение |
|---|---:|---|
| `quiet` | none | baseline |
| `farm_light` | 20 dB | реалистичный легкий фон |
| `farm_medium` | 12 dB | сложный, но не разрушительный фон |

Не стоит начинать с сильного шума ниже 8 dB: на маленьком датасете это даст много неинформативных ошибок.

## Ручная валидация

Для каждой строки нужно проверить:

- аудио соответствует `golden_transcript`;
- голос русский, без заметного иностранного акцента;
- бирки/коды/препараты произнесены так, как пользователь реально сказал бы;
- шум не перекрывает ключевые entity;
- разметка `number / proper_noun / other` корректна;
- entity-level поля не противоречат tool call.

## Артефакты в репозитории

- private ASR manifest - 72 выбранные фразы и разметка.
- private ASR audio directory - целевая папка для WAV.
- private ASR summary - распределение по стратам/голосам/шуму.
- `scripts/generate_asr_dataset_manifest.py` - генератор manifest.
- `scripts/evaluate_asr_predictions.py` - WER/entity WER/tag exact-match.
- `scripts/synthesize_asr_audio_yandex.py` - генерация WAV через Yandex SpeechKit и deterministic noise overlay.

## Следующий шаг

Перед генерацией аудио нужно выбрать провайдера:

1. Yandex SpeechKit как основной baseline.
2. SaluteSpeech как русскоязычная альтернатива для 10-15 контрольных фраз.
3. ElevenLabs или Azure только если нужно больше разнообразия голосов.

Yandex-провайдер уже подключен через `scripts/synthesize_asr_audio_yandex.py`. Для повторной генерации:

```bash
export YANDEX_API_KEY=<api-key>
python3 scripts/synthesize_asr_audio_yandex.py --force
```
