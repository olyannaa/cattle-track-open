# Codex Workflow

## 1. Постановка задачи

Перед началом работы Codex читает ближайший код и документы, затем уточняет только то, что нельзя безопасно вывести из контекста. Для технических задач приоритет у существующей архитектуры проекта.

## 2. Изменения

Codex вносит изменения небольшими связными шагами:

- читает текущую реализацию;
- правит только затронутые файлы;
- не откатывает чужие изменения;
- не добавляет новые зависимости без необходимости;
- документирует новые процессы, если они важны для запуска или поддержки проекта.

## 3. Проверки

Минимальный набор выбирается по зоне изменения:

- frontend: `npm run build`, при необходимости браузерная проверка;
- landing: Docker-сборка `docker compose up -d --build landing`;
- backend: `dotnet test backend/CAT.Tests/CAT.Tests.csproj`;
- AI-контракты: `./scripts/verify-ai.sh`;
- полный технический контур: `./scripts/verify-all.sh`.

Для UI-изменений Codex дополнительно проверяет фактическое отображение через браузерные метрики или скриншоты, а не только компиляцию.

## 4. Git

После успешной проверки Codex делает осмысленный commit и push в рабочую ветку. В сообщении пользователю указываются:

- что изменено;
- какие проверки прошли;
- commit hash;
- оставшиеся ограничения, если они есть.

## 5. Ожидаемый уровень качества

Работа с агентом считается завершенной только если:

- изменение воспроизводимо из git;
- не осталось незакоммиченного diff;
- пользовательский сценарий проверен;
- документация обновлена, если менялся запуск, процесс или AI-контур.
