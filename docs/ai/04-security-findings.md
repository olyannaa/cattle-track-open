# Находки по безопасности API (отдельно от AI-проекта)

Обнаружено при анализе кода backend. Существует в проде независимо от AI-темы, чинится отдельным тикетом. Не блокирует AI-модуль, но AI-тулы, затрагивающие эти эндпоинты, не должны наследовать дефекты.

## 1. DELETE групп/типов/полей идентификации без org-проверки
`DELETE /api/groups`, `/api/groups/type`, `/api/groups/identification` — нет `[OrgValidationTypeFilter]` (в отличие от Create/Edit/Get тех же контроллеров). Возможно удаление чужой сущности по угаданному Guid.
**Фикс:** добавить `[OrgValidationTypeFilter(checkOrg: true)]`.

## 2. Незащищённый дев-эндпоинт удаления пользователя
`DELETE /api/Users` (комментарий «ДЛЯ РАЗРАБОТКИ») — без `[Authorize]`, удаляет пользователя по login анонимно.
**Фикс (критично):** убрать из прод-ветки либо закрыть авторизацией. Не оборачивать в тул ни при каких условиях.

## 3. `UpdateEmployee` — избыточный контракт
`POST /api/organizations/users/{userId}` принимает `Login`/`Password`/`RoleId` разом — «смена роли» технически позволяет сменить чужой логин/пароль.
**Фикс:** разделить на узкие эндпоинты (ChangeRole, ResetPassword) с минимальными DTO.

## 4. Нет confirm на удаление/увольнение сотрудника (frontend)
В отличие от групп (есть `ConfirmModal`), удаление пользователя и смена роли — без подтверждения.
**Фикс:** единый confirm-паттерн.

---

Завести отдельными задачами в трекере компании. По возможности поправить п.1 до открытия AI-тулов `delete_group`/`delete_group_type`/`delete_identification_field` (иначе агент унаследует брешь авторизации — до фикса org-проверка обязательна на стороне AI-сервиса, см. `00-tool-catalog.md` §6).
