# AI Assistant MVP Use Cases

Дата: 2026-07-14

Цель документа: зафиксировать пользовательские потоки AI-помощника Cattle Track до дальнейшей доработки выбора животных, preview и confirm. Помощник должен быть единым диалоговым окном, которое открывается только по кнопке AI-ассистента. На страницах учета не должно быть встроенного чата.

## Общие правила интерфейса

1. AI-помощник открывается только из глобальной кнопки.
2. На страницах форм не рендерится отдельный экземпляр `AiEventInput`.
3. Главный сценарий ввода - голосовой. Текстовый ввод остается для правок, уточнений и ручного ввода.
4. Любая write-операция сначала создает preview. Запись в БД разрешена только после явного подтверждения.
5. Если найдено несколько животных с одной биркой, система показывает карточки-кандидаты и кнопку выбора. Пользователь не должен заново писать тег, если он уже выбирает из списка.
6. Если пользователь уточняет данные во втором сообщении, система обязана учитывать историю текущего диалога.
7. Технические ошибки схемы, DTO или tool call не показываются пользователю. Пользователь видит человеческий текст: что найдено, что не найдено, чего не хватает, что можно сделать дальше.

## Entity Resolution

### AnimalRef

Пользователь может назвать:

- бирку: `25`, `523`, `1432`;
- фразу с типом животного: `корова 25`, `бык 77`;
- дополнительный идентификатор;
- дату рождения;
- контекстную ссылку: `эту корову`, `ту первую`, `про нее`.

Backend resolver:

- ищет бирку точным совпадением внутри `OrganizationId`;
- не использует fuzzy-поиск как финальный поиск по БД;
- нормализует безопасные LLM-префиксы: `cow_25`, `корова 25` -> `25`;
- возвращает одно из состояний: `resolved`, `ambiguous`, `not_found`.

Почему без fuzzy-поиска по БД:

- в хозяйстве могут намеренно существовать животные с одинаковыми или очень похожими бирками;
- fuzzy-поиск по БД может выбрать не то животное и привести к записи события не в ту карточку;
- для write-операций это особенно опасно: вес, выбытие, лечение, осеменение и беременность меняют состояние системы;
- fuzzy допустим только до resolver-а как коррекция ASR/текста, например распознанное `пятьсот двадцать три` -> `523`;
- после нормализации resolver всё равно ищет точное значение бирки в рамках организации.

### Ambiguous Candidates

Если найдено 2-5 животных:

- показать все варианты карточками;
- на карточке показать бирку, статус, тип, дату рождения, группу, породу, дополнительные идентификаторы;
- каждая карточка должна иметь действие `Выбрать`;
- после выбора система должна продолжить исходный сценарий, а не отправлять новый LLM-запрос с текстом `Выбрано...`.

Если найдено больше 5 животных:

- показать первые 5;
- попросить уточнить дату рождения, группу или дополнительный идентификатор;
- не выполнять write commit.

## Поток 1. Найти животное

Примеры:

- `найди корову 25`
- `покажи животное с биркой 523`
- `открой быка 77`

### 0 совпадений

Система отвечает: животное с такой биркой не найдено в организации. Предлагает проверить бирку или назвать дополнительный идентификатор.

Действий с БД нет.

### 1 совпадение

Система показывает краткую карточку животного в чате:

- бирка;
- тип;
- статус;
- дата рождения;
- группа;
- порода;
- дополнительные идентификаторы.

Если есть ссылка/действие открытия полной карточки, пользователь может перейти в привычный раздел карточки животного.

### 2-5 совпадений

Система показывает список кандидатов с кнопками `Выбрать`.

После выбора:

- система показывает карточку выбранного животного;
- не требует заново вводить tag;
- сохраняет выбранное животное в session context.

## Поток 2. Получить карточку животного

Примеры:

- `покажи карточку коровы 25`
- `открой карточку животного 523`
- `что по корове 1432`

Логика совпадает с потоком поиска, но при resolved состоянии ответ должен быть карточкой, а не просто строкой `нашла животное`.

### 2-5 совпадений

После выбора кандидата:

- система должна вызвать получение карточки по выбранному `animal_id`;
- не должна повторно просить `tag`;
- не должна отправлять в LLM текст `Выбрано: бирка 25`.

## Поток 3. Родители животного

Примеры:

- `покажи родителей коровы 25`
- `кто мать и отец у 523`

### 0 совпадений

Показать человеку понятный ответ: `Животное с биркой X не найдено в вашей организации. Проверьте бирку или назовите дополнительный идентификатор.`

### 1 совпадение

Backend делает multi-hop:

1. resolved animal;
2. получить `motherId` и `fatherIds`;
3. получить бирки родителей;
4. ответить текстом.

### 2-5 совпадений

Сначала показать выбор животного.

После выбора:

- выполнить multi-hop по выбранному `animal_id`;
- показать родителей;
- сохранить выбранное животное в session context.

## Поток 4. История веса

Примеры:

- `покажи историю веса коровы 25`
- `какой вес был у 523`

### 0 совпадений

Показать человеку понятный ответ: `Животное с биркой X не найдено в вашей организации. Проверьте бирку или назовите дополнительный идентификатор.`

### 1 совпадение

Показать историю веса:

- дата;
- вес;
- метод;
- последние N записей.

### 2-5 совпадений

Показать кандидатов.

После выбора:

- показать историю веса выбранного животного;
- не требовать повторный tag.

## Поток 5. Список групп

Примеры:

- `покажи группы`
- `какие есть группы животных`

Resolver не нужен.

Система сразу показывает список групп. Если групп много, ответ должен быть кратким для голоса и полным/сканируемым для текста.

## Поток 6. Коровы для проверки стельности

Примеры:

- `кого нужно проверить на стельность`
- `покажи коров для диагностики беременности`

Resolver конкретного животного не нужен.

Система сразу показывает список коров для проверки. Ответ должен быть read-only.

## Поток 7. Внести вес

Примеры:

- `запиши вес 421 кг бирка 25 сегодня ручное взвешивание`
- `добавь вес 423 кг корове 25 за 3 января 2026 ручное взвешивание`

Обязательные данные:

- животное;
- вес;
- дата;
- метод взвешивания.

### Не хватает обязательных данных

Система не создает commit-ready preview. Она отвечает, чего не хватает:

- `Нужно указать вес`;
- `Нужно указать дату`;
- `Нужно выбрать метод взвешивания`;
- `Нужно выбрать животное`.

Пользователь может дать недостающую информацию следующим сообщением. Сессия должна помнить уже сказанные данные.

### 0 совпадений животного

Показать человеку понятный ответ: `Животное с биркой X не найдено. Запись не создана. Проверьте бирку или назовите дополнительный идентификатор.`

### 1 совпадение

Создать draft preview:

- бирка;
- вес;
- дата;
- метод;
- кнопки `Отменить` и `Подтвердить`;
- возможность голосом или текстом сказать `Подтверждаю`, `Сохрани`, `Да, записывай`, `Отмени`, `Не сохраняй`.

До confirm записи в БД нет.

### 2-5 совпадений

Показать кандидатов.

После выбора:

- пересобрать тот же draft с выбранным `animal_id`;
- если все обязательные поля есть, показать preview;
- если чего-то не хватает, попросить только недостающее поле.

### Confirm

Commit идет через существующий backend service веса.

Ответ: `Информация внесена: запись о весе для животного с биркой X.`

Confirm может быть выполнен:

- нажатием кнопки `Подтвердить`;
- голосовой командой `Подтверждаю`;
- текстовой командой `Подтверждаю`, `Сохрани`, `Да, записывай`.

### Cancel

Draft отменяется.

Ответ: `Черновик отменён.`

Cancel может быть выполнен:

- нажатием кнопки `Отменить`;
- голосовой командой `Отмени`;
- текстовой командой `Отмени`, `Не сохраняй`, `Отмена`.

## Поток 8. Daily Action

MVP daily action включает:

- осмотр;
- обработку;
- вакцинацию/обработку;
- лечение;
- перевод;
- выбытие;
- исследование;
- присвоение номеров;
- изменение половозрастной группы.

Примеры:

- `добавь лечение корове 25 сегодня`
- `переведи 523 в группу нетели сегодня`
- `поставь вакцинацию 25 препаратом X дозировка Y`

Обязательные данные зависят от типа действия.

### Single item

Сценарий такой же, как у веса:

1. entity resolution;
2. validation;
3. preview;
4. confirm/cancel.

### Batch

Пример:

- `добавь обработку коровам 25, 26 и 27 сегодня`

Batch делится на:

- `resolved`;
- `ambiguous`;
- `not_found`;
- `invalid`.

Система показывает partial preview. Нельзя тихо терять невалидные элементы.

Если часть животных ambiguous:

- показать кандидатов по каждому ambiguous item;
- пользователь выбирает нужных животных;
- после каждого выбора пересчитать preview.

Confirm сохраняет только commit-ready items. Остальные должны быть явно показаны как skipped/failed.

## Поток 9. Осеменение

Примеры:

- `внеси осеменение корове 25 сегодня естественное`
- `осемени 523 сегодня искусственное, партия спермы ABC`

Обязательные данные:

- корова;
- дата;
- тип осеменения.

Условные данные:

- для искусственного: партия спермы или другие поля, которые требует существующая форма/сервис;
- для естественного: бык, если бизнес-правило требует его;
- для эмбриона: данные эмбриона, если бизнес-правило требует их.

### 0 совпадений

Показать человеку понятный ответ: `Животное с биркой X не найдено. Осеменение не подготовлено. Проверьте бирку или назовите дополнительный идентификатор.`

### 1 совпадение

Создать draft preview. Не писать в БД до confirm.

### 2-5 совпадений

Показать кандидатов.

После выбора:

- продолжить тот же draft осеменения;
- не требовать повторный tag;
- если данных достаточно, показать preview;
- если данных недостаточно, спросить только недостающее.

### Batch

Batch осеменения возможен только если backend и форма реально поддерживают batch для этого действия. Если форма/сервис не поддерживает batch для конкретного сценария, ассистент должен предложить внести по одной корове.

## Поток 10. Уточнение во втором сообщении

Пример:

1. Пользователь: `внеси осеменение корове 25`
2. Система: `Нужно уточнить дату и тип осеменения.`
3. Пользователь: `сегодня естественное`

Ожидаемое поведение:

- система использует корову `25` из предыдущего сообщения;
- не спрашивает бирку заново;
- создает preview осеменения;
- если коров с биркой 25 несколько, сначала показывает выбор, затем применяет уточнение.

## Поток 11. Исправление и отмена

### Исправление до confirm

Пример:

1. `запиши вес 421 кг корове 25 сегодня`
2. preview;
3. `нет, вес 412`

Ожидаемое поведение:

- обновить draft;
- снова показать preview;
- не делать commit до подтверждения.

### Отмена

Пример:

- `отмени`
- `не сохраняй`
- пользователь нажал `Отменить`.
- пользователь сказал голосом `отмени`.

Ожидаемое поведение:

- draft становится canceled;
- confirm после отмены невозможен;
- ответ: `Черновик отменён.`

## Что нужно исправить перед ошибкой 2

1. Сделать единый механизм выбора кандидата:
   - read-flow: selection продолжает read tool по `animal_id`;
   - write-flow: selection обновляет draft item и пересчитывает preview.
2. Не отправлять выбор кандидата как обычное сообщение пользователя в LLM.
3. У candidate card должна быть машинная связь:
   - `draftId` или `pendingResolutionId`;
   - `itemIndex`, если это write item;
   - `candidateId`;
   - исходный tool/action.
4. После выбора read-кандидата нельзя снова требовать `tag`.
5. После выбора write-кандидата должна появляться кнопка confirm, если все остальные обязательные поля валидны.
6. UI должен различать:
   - `read ambiguity`;
   - `write draft ambiguity`;
   - `invalid missing fields`;
   - `commit-ready preview`.
